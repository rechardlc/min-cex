# Golang CEX / Web3 后端中高级面试手册

这份文档基于 `GOLANG_CEX_BACKEND_ROADMAP.md`，把学习路线中的技术栈和业务模块转成中高级面试内容。

面试目标不是背框架 API，而是证明你具备以下能力：

```text
理解 CEX / Web3 后端核心业务链路
能设计账户、订单、账本、撮合、钱包、行情等模块
能解释事务、幂等、事件驱动、链上确认等关键问题
能分析性能瓶颈
能设计兜底方案
能排查线上问题
能在故障中保护资产安全和数据一致性
```

## 2. 项目总览类问题

### Q1：请介绍你的 Mini CEX 后端项目

回答结构：

```text
项目目标：
  用 Go 实现一个最小 CEX 后端，覆盖账户、订单、撮合、结算、行情、充值、提现。

技术栈：
  go-zero、PostgreSQL、Redis、NATS、go-ethereum、Anvil、Docker Compose。

核心模块：
  API 服务、账户服务、订单服务、撮合服务、结算服务、行情服务、钱包扫描服务、提现服务。

核心链路：
  用户充值 -> 入账 -> 下单冻结资金 -> 撮合成交 -> 账本结算 -> 行情推送 -> 提现广播。

重点能力：
  事务、幂等、事件驱动、链上确认、故障恢复、测试和压测。
```

加分表达：

```text
这个项目不是单纯 CRUD，我重点模拟的是 CEX 的资产安全和交易链路。
比如下单前冻结资金，成交后异步结算，充值扫描需要确认数和幂等，提现要有状态机和重试机制。
```

## 3. Go 后端工程面试

### Q1：go-zero 在项目中承担什么角色？

回答要点：

```text
go-zero 负责 API 和 RPC 服务的工程结构。
API 层处理外部 HTTP 请求。
zrpc 层处理内部服务调用。
svc 层管理依赖。
logic 层承载业务逻辑。
配置、日志、中间件也由框架统一管理。
```

Web3 / CEX 场景：

```text
api-service：
  下单、查账户、查订单、查充值提现。

account-rpc：
  冻结资金、释放资金、更新账本。

order-rpc：
  创建订单、撤单、更新订单状态。

wallet-rpc：
  处理充值入账、提现状态更新。
```

性能关注：

```text
避免 API handler 中做重逻辑。
RPC 调用必须设置 timeout。
服务之间要有错误码和降级策略。
高频接口要做限流和缓存。
```

兜底方案：

```text
RPC 超时：
  返回明确错误，不要无限等待。

下游账户服务不可用：
  下单接口直接拒绝，不能先接单后冻结失败。

行情接口压力过大：
  优先返回 Redis 缓存快照。
```

### Q2：Go 服务出现 goroutine 泄漏怎么排查？

回答步骤：

```text
1. 看 goroutine 数量是否持续增长。
2. 使用 pprof /debug/pprof/goroutine 查看堆栈。
3. 定位卡住的位置，是 channel 阻塞、锁阻塞、网络请求无超时，还是 ticker 没释放。
4. 检查 context 是否正确传递和取消。
5. 检查 worker 是否有退出机制。
```

CEX 场景：

```text
wallet-scanner 长期运行，如果每个区块扫描都开 goroutine 但没有回收，会导致泄漏。
WebSocket 每个连接对应 goroutine，如果断连没有清理，也会泄漏。
```

兜底：

```text
所有外部请求必须带 context timeout。
ticker 使用后 defer Stop。
channel 消费要有退出信号。
WebSocket 断开后清理订阅关系。
```

## 4. 数据库与账本面试

### Q1：CEX 为什么要区分 available_balance 和 frozen_balance？

回答要点：

```text
available_balance：
  用户可自由使用的余额。

frozen_balance：
  已被订单、提现、风控等业务占用的余额。
```

业务例子：

```text
用户挂买单：
  冻结 USDT。

用户挂卖单：
  冻结 ETH。

用户撤单：
  释放未成交部分冻结资产。

订单成交：
  扣减冻结资产，增加对手资产。
```

性能关注：

```text
账户表更新是高频热点。
需要按 user_id + asset 精确更新。
要有合适索引。
事务范围要短。
避免在事务中做 RPC / MQ / 链上请求。
```

兜底方案：

```text
余额更新失败：
  整个事务回滚。

重复请求：
  business_id 唯一索引防重。

余额不足：
  直接拒绝，不进入撮合。
```

### Q2：如何保证余额变更一致性？

回答结构：

```text
1. 所有余额变更必须走账户服务。
2. 使用数据库事务。
3. 使用行锁锁定 user_id + asset 账户行。
4. 检查余额不能为负。
5. 更新账户余额。
6. 写 account_ledgers 流水。
7. 使用 business_type + business_id 做幂等。
8. 提交事务。
```

示例 SQL 思路：

```sql
SELECT * FROM accounts
WHERE user_id = $1 AND asset = $2
FOR UPDATE;
```

面试加分点：

```text
余额表是当前状态，流水表是审计记录。
不能只更新余额不写流水。
流水需要能反推资产变化原因。
```

排查问题：

```text
如果用户反馈余额不对：
  先查账户当前余额。
  再查 account_ledgers。
  按 business_id 关联订单、成交、充值、提现。
  检查是否有重复入账或漏入账。
  检查事务是否有失败但事件被确认消费。
```

### Q3：如何避免重复充值入账？

回答要点：

```text
充值事件唯一标识：
  chain_id + tx_hash + log_index

在 chain_events 或 deposits 表上建立唯一索引。
扫描到事件后先落 event_log。
确认数满足后执行入账。
入账时使用 deposit_id / event_id 做业务幂等。
```

兜底方案：

```text
重复扫到同一个 event：
  unique key 拦截。

入账过程中服务重启：
  根据状态继续处理 CONFIRMED 但未 CREDITED 的充值。

RPC 返回不稳定：
  多次确认，按区块高度和 tx receipt 校验。
```

排查问题：

```text
用户说充值未到账：
  查 tx 是否成功。
  查区块确认数。
  查 scanner last_scanned_block。
  查 chain_events 是否记录。
  查 deposits 状态。
  查 account_ledgers 是否有 CREDITED 流水。
```

## 5. 订单与撮合面试

### Q1：限价单什么时候挂单，什么时候成交？

回答要点：

```text
买入限价单：
  如果 best ask <= buy price，说明可以买到更便宜或等价的卖单，应该立即撮合。
  剩余未成交部分再挂入买盘。

卖出限价单：
  如果 best bid >= sell price，说明可以买方愿意出更高或等价价格，应该立即撮合。
  剩余未成交部分再挂入卖盘。
```

核心规则：

```text
价格优先
时间优先
同价格 FIFO
```

性能关注：

```text
获取 best bid / best ask 必须快。
订单 ID 查询必须 O(1) 或接近 O(1)。
撤单不能遍历整个订单簿。
价格档需要高效插入、删除、查最优价。
```

兜底：

```text
撮合前订单已经被取消：
  命令序列化处理，避免并发状态错乱。

订单部分成交后失败：
  撮合事件要可重放，结算要幂等。
```

### Q2：撮合引擎为什么通常单线程处理一个交易对？

回答要点：

```text
撮合最重要的是顺序确定性。
同一个交易对的订单必须严格按进入顺序处理。
单线程事件循环可以避免复杂锁竞争。
结果可重放、可审计。
延迟更稳定。
```

性能扩展方式：

```text
一个交易对一个 event loop。
多个交易对并行。
BTC-USDT 一个撮合实例。
ETH-USDT 一个撮合实例。
不同 symbol 之间天然可并行。
```

兜底方案：

```text
撮合服务重启：
  从订单快照 + 事件日志恢复。

撮合积压：
  限流入口。
  按交易对隔离。
  监控 command queue lag。
```

排查问题：

```text
如果用户说订单顺序不对：
  查订单进入队列时间。
  查撮合命令序列号。
  查同价格档 FIFO 队列。
  查是否有并发路径绕过撮合事件循环。
```

### Q3：市价单流动性不足怎么办？

回答方式：

```text
这取决于产品规则。
可以选择全部拒绝，也可以允许部分成交。
```

常见策略：

```text
全部拒绝：
  如果订单簿对手方总量不足，直接 reject。

部分成交：
  吃掉可用流动性，剩余部分取消。

带保护价格：
  市价单设置最大滑点，超过保护价格停止成交。
```

兜底：

```text
不能 panic。
必须返回明确错误码，例如 INSUFFICIENT_LIQUIDITY。
必须记录 reject reason，方便用户和客服查询。
```

## 6. 结算与事件驱动面试

### Q1：为什么撮合和结算要拆开？

回答要点：

```text
撮合追求低延迟和顺序确定性。
结算涉及数据库事务、账本、手续费、幂等，可能较慢。
如果撮合里直接做数据库和链上操作，会拖慢撮合核心。
```

架构：

```text
matching-engine
  -> TradeEvent
  -> settlement-service
  -> update orders / accounts / ledgers
```

兜底：

```text
TradeEvent 重复消费：
  trade_id 唯一索引保证幂等。

结算失败：
  事件重试。
  进入 dead letter。
  人工补偿。

结算积压：
  监控 lag。
  扩容 settlement worker。
  按 symbol 或 trade_id 分片。
```

### Q2：NATS / Kafka 在交易系统里解决什么问题？

回答要点：

```text
解耦服务。
异步处理。
削峰。
事件广播。
失败重试。
让撮合、结算、行情、钱包等模块边界清晰。
```

NATS 与 Kafka 区别：

```text
NATS：
  轻量，低延迟，部署简单，适合训练项目和中小规模事件。

Kafka：
  高吞吐，持久化强，支持回放，适合大规模事件流和审计场景。
```

排查问题：

```text
如果结算延迟：
  看消息堆积。
  看 consumer 是否报错。
  看数据库事务耗时。
  看是否有热点账户锁等待。
  看是否有重复消息导致频繁冲突。
```

## 7. Web3 链上交互面试

### Q1：ERC20 充值扫描怎么设计？

回答结构：

```text
1. 配置 chain_id、contract_address、start_block。
2. scanner 定时获取最新区块高度。
3. 按 block range 调 eth_getLogs。
4. 过滤 Transfer event。
5. 判断 to 地址是否是平台充值地址。
6. 写入 chain_events / deposits。
7. 等待确认数。
8. 确认后调用账户服务入账。
9. 更新 deposit 状态为 CREDITED。
10. 记录 last_scanned_block。
```

性能关注：

```text
不能一次扫太大区间。
需要控制 RPC 请求频率。
高峰期按区块范围分批。
event 表要有 tx_hash + log_index 唯一索引。
充值地址查询要有索引或缓存。
```

兜底：

```text
RPC 失败：
  重试，切换备用 RPC。

服务重启：
  从 last_scanned_block 继续。

链重组：
  保留确认数，例如 12 个块后再入账。
  或者回扫最近 N 个区块。

重复事件：
  唯一索引去重。
```

### Q2：提现系统怎么设计？

状态机：

```text
CREATED
APPROVED
BROADCASTING
BROADCASTED
CONFIRMED
FAILED
```

流程：

```text
1. 用户申请提现。
2. 冻结提现资产。
3. 风控审核。
4. wallet-worker 获取待广播任务。
5. 构造交易。
6. 签名。
7. 广播。
8. 保存 tx_hash。
9. 查询 receipt。
10. confirmed 后扣减 frozen_balance。
11. failed 时根据规则释放或重试。
```

性能关注：

```text
提现不是追求极致高 QPS，而是追求安全和可靠。
nonce 管理是关键。
广播任务要串行或按钱包地址分片处理。
```

兜底：

```text
广播失败：
  记录错误，重试或进入人工处理。

交易 pending 太久：
  检查 gas 是否过低。
  支持加速交易。

nonce 冲突：
  按 from_address 管理 nonce。
  同一个热钱包地址的广播任务串行化。

receipt failed：
  根据链上结果更新提现失败，并决定是否释放冻结资产。
```

排查问题：

```text
用户说提现不到账：
  查 withdrawal 状态。
  查 tx_hash。
  查链上 receipt。
  查 nonce 是否卡住。
  查热钱包余额和 gas。
  查 RPC 是否返回异常。
  查是否被风控拦截。
```

### Q3：链上确认数为什么重要？

回答要点：

```text
区块链可能发生短暂 reorg。
如果 0 确认就入账，可能遇到交易被回滚。
确认数是用延迟换安全。
```

策略：

```text
普通 ERC20：
  可以设置 12 确认。

高价值充值：
  可以设置更高确认数。

不同链：
  根据出块时间、finality、历史稳定性配置不同确认数。
```

## 8. Redis、缓存与限流面试

### Q1：Redis 在 CEX 后端里用在哪里？

回答要点：

```text
行情快照缓存
交易对配置缓存
用户 token / session 辅助
API 限流
短期幂等 key
WebSocket 订阅状态
热点数据缓存
```

不能做什么：

```text
不能只用 Redis 存用户资产最终余额。
账本最终状态必须落数据库。
Redis 可以加速，但不能替代核心账本。
```

兜底：

```text
Redis 不可用：
  核心下单和账本不能依赖 Redis 才能保证正确性。
  缓存失败最多降级为查数据库。
  限流失败可以选择保守拒绝或降级限流。
```

### Q2：如何设计下单接口限流？

回答要点：

```text
按 user_id 限流。
按 API key 限流。
按 IP 限流。
按交易对限流。
不同接口不同额度。
```

实现：

```text
Redis key:
  rate:{user_id}:{endpoint}:{window}

使用 INCR + EXPIRE
或 token bucket / sliding window。
```

兜底：

```text
限流服务异常时：
  对交易接口可以保守拒绝。
  对查询接口可以降级。
```

## 9. 行情与 WebSocket 面试

### Q1：行情服务如何设计？

回答结构：

```text
撮合服务产生 TradeEvent / BookEvent。
行情服务消费事件。
维护 orderbook snapshot。
维护 recent trades。
计算 ticker。
通过 WebSocket 推送给订阅用户。
HTTP 查询走 Redis 或内存快照。
```

性能关注：

```text
WebSocket 连接数。
慢客户端。
消息广播 fanout。
单连接写阻塞。
消息序列号。
快照 + 增量一致性。
```

兜底：

```text
客户端断线：
  重连后先拉快照，再订阅增量。

消息丢失：
  使用 sequence 检测。
  缺口时重新拉快照。

慢客户端：
  写队列满则断开连接，避免拖垮服务。
```

排查问题：

```text
用户说行情不动：
  看撮合是否产生 BookEvent。
  看 NATS/Kafka 是否有消息。
  看 marketdata consumer 是否正常。
  看 WebSocket 连接是否还活着。
  看客户端订阅 topic 是否正确。
```

## 10. 性能专题

### 10.1 下单接口性能瓶颈

可能瓶颈：

```text
参数校验
鉴权
Redis 限流
数据库冻结资金事务
订单落库
RPC 调用账户服务
消息队列 publish
```

优化方向：

```text
缩短数据库事务时间。
账户冻结 SQL 精确命中索引。
热点用户或机器人限流。
下单进入队列后异步撮合。
避免在下单请求里等待结算完成。
```

排查：

```text
看 API p95 / p99。
看 DB 慢查询。
看锁等待。
看 MQ publish 延迟。
看账户服务 RPC 延迟。
```

### 10.2 撮合性能瓶颈

可能瓶颈：

```text
价格档查找慢
撤单遍历订单簿
每次获取 best price 都排序
事件输出阻塞
锁竞争
GC 压力
```

优化方向：

```text
维护 best bid / best ask。
订单 ID -> order 指针索引。
价格 -> price level 索引。
同价格订单使用 FIFO 队列。
撮合核心单线程事件循环。
事件输出异步化。
减少临时对象分配。
```

排查：

```text
用 benchmark 测撮合吞吐。
用 pprof 看 CPU 和 allocation。
用队列长度看命令积压。
用延迟指标看 p99。
```

### 10.3 数据库性能瓶颈

可能瓶颈：

```text
账户表热点行锁
```

## 11. 兜底方案专题

### 11.1 下单兜底

```text
账户服务不可用：
  拒绝下单，不能先接单。

余额不足：
  REJECTED，记录原因。

订单消息发送失败：
  订单保持创建失败或进入待发送状态，由补偿任务处理。

撮合服务积压：
  限流、暂停部分交易对、扩容或降级查询。
```

### 11.2 结算兜底

```text
重复 TradeEvent：
  trade_id 唯一索引。

结算失败：
  重试 + dead letter + 人工补偿。

账户更新失败：
  事务回滚。

部分订单已更新但余额失败：
  不允许跨事务拆开核心状态。
```

### 11.3 充值兜底

```text
RPC 失败：
  切换备用 RPC。

scanner 重启：
  从 last_scanned_block 继续。

链重组：
  确认数 + 最近区块回扫。

重复事件：
  tx_hash + log_index 唯一索引。
```

### 11.4 提现兜底

```text
nonce 卡住：
  按地址串行广播，检查 pending tx。

gas 过低：
  支持加速或重发。

广播后未拿到 tx_hash：
  查询本地 nonce 和链上 pending，避免重复扣款。

receipt failed：
  标记失败，按规则释放冻结或人工处理。

热钱包余额不足：
  提现任务暂停，告警，等待归集或补 gas。
```

## 12. 排查问题能力专题

### 场景 1：用户说下单成功但订单没成交

排查链路：

```text
1. 查订单状态。
2. 查订单是否进入撮合队列。
3. 查撮合服务是否消费到命令。
4. 查订单价格是否 crossing book。
5. 查对手盘深度。
6. 查撮合事件是否生成。
7. 查结算是否完成。
8. 查 WebSocket 是否推送。
```

### 场景 2：用户说成交了但余额没变

排查链路：

```text
1. 查 trade 是否存在。
2. 查 settlement 状态。
3. 查 trade_id 是否已结算。
4. 查 account_ledgers。
5. 查账户余额。
6. 查结算服务错误日志。
7. 查消息队列是否积压。
```

### 场景 3：充值长时间不到账

排查链路：

```text
1. 查链上 tx 是否成功。
2. 查确认数。
3. 查 scanner 当前高度。
4. 查 event_log 是否记录。
5. 查 deposit 状态。
6. 查充值地址是否属于平台。
7. 查是否满足最小充值金额。
8. 查入账流水。
```

### 场景 4：提现 pending 很久

排查链路：

```text
1. 查 withdrawal 状态。
2. 查 tx_hash。
3. 查 receipt。
4. 查 gas fee 是否过低。
5. 查 nonce 是否被前序交易卡住。
6. 查热钱包余额。
7. 查 RPC 状态。
8. 决定加速、等待或人工处理。
```

### 场景 5：行情 WebSocket 大面积断开

排查链路：

```text
1. 看连接数变化。
2. 看 WebSocket 服务 CPU / memory。
3. 看写队列是否堆积。
4. 看是否有慢客户端。
5. 看 NATS/Kafka 消费是否正常。
6. 看网络和负载均衡。
7. 看最近是否发布大消息或异常广播。
```

## 14. 面试回答原则

回答中高级问题时，尽量按这个结构：

```text
1. 先说业务目标。
2. 再说核心数据结构或状态机。
3. 再说正常流程。
4. 再说异常场景。
5. 再说幂等和一致性。
6. 再说性能瓶颈。
7. 最后说监控和兜底。
```

示例：

```text
充值入账不是简单扫到 Transfer 就加钱。
正常流程是扫描 event、去重、等待确认数、入账、写流水。
异常上要处理 RPC 失败、链重组、重复事件、服务重启。
一致性上用 tx_hash + log_index 唯一索引和 deposit 状态机。
性能上控制扫描区间和 RPC 频率。
监控上看 scanner block lag、deposit pending 数和 RPC 错误率。
```
